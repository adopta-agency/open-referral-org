I am Kin Lane, the API Evangelist...

This is my work in partnership with Open Referral to move forward the Human Services Data Specification (HSDS), in conjunction with a definition for the Human Services Data API (HSDA). I've dedicated a portion of my time as the API Evangelist to help move forward the conversation about how we store, share, publish, and collaborate around human services data.

I began this work with Greg Bloom of Open Referral back in 2015 and 2016, and signed on as the technical lead by 2017. I'm invested in the projects success, and will invest as much time as I possibly can helping contribute to the data schema, API definition, tooling, an dstorytelling. I feel the project reflects the API potential, helping make our world work better for humans, not just something technical that startups do.

What Is The Human Services Data Specification

The vision of Open Referral is a world where everyone has access to — and the ability to effectively use — information about the resources that are available in their community.

The Human Service Data Specification (HSDS) is a data schema used to describe human services that are present in any city, state, or province. The schema is to provide a common way to descibe information, helping standardize how it can be stored, and shared in service of the Open Referral vision.

The Human Services Data API (HSDA) is an OpenAPI definition providing the technical details of an API which can provide access to data stored in the HSDS format. An API will open up human service systems and websites for integraiton with other systems, web and mobile applications--helping ensure everyone has access to, and the ability to put human services to work.

Human Service Providers / Implementations

This work is not about providing a single API. It is about providing a blueprint for providing many APIs. The schema and API definition is meant to provide a common blueprint that any government or human services organization can follow to design and deploy their own HSDS complaint API ensuring it will speak to all the other HSDS compliant APIs--making human service websites and application interoperable.

I'm supporting Open Referral's vision of "a world where everyone has access to — and the ability to effectively use — information about the resources that are available in their community." Embracing the open web, and acknowledging that humans and other systems and applications will benefit from a common data schema, as well as API definition, when it comes to individual human services implementations, as well as interoperability, aggregation, and federation of data across potentially thousands of individual implementations.

Vendors Serving The Community

This work is not ignoring the fact that there are already numerous software providers already providing the humn services data solutons in use across 211, and other municipal and health care systems. We are actively working with the top vendors to help gather their feedback, and expertise in operating real world human service implementations, and incorporate in future versions of the specification.

We are currently taking the schema and API documentation from over five providers and including as part of the current schema and API release. We will be actively working with new providers as they approach Open Referral. Our goal is to work with as many of the leadings providers as we can, and ensure interoperability with all the major systems that are currently serving municpalities.

For the Human Services Data Specification to be successful it will need to have the buy-in of major commercial providers, making it an industry standard in use across human services implementations worldwide.

Focus On Platform Projects

Heroku

The original Ohana implementation of Open Referral runs on Heroku, and we are looking to develop additional implentations that will deploy and operate via the Heroku cloud platform, and even potentially connect with SalesForce along the way.

Amazon Web Services

The Human Services API prototype in PHP currently runs on Amazon leveraging EC2, S3, and RDS for the backend. We are looking to develop additional solutions that run on AWS, and leverage a variety of AWS solutions like Labmda, API Gateway, and more.

Azure

We have begun to explore what it would take to design, deploy, and manage human services API implementations on the Azure platform. We'd like to have a working prototype, as well as specific implementations on Azure that we can showcase and help support future efforts on the cloud provider.

Google

We have also begun explore what it would take to design, deploy, and manage human service API implementations on the Google platform. We are looking for investment, developers, and potential implementations that would benefit from running on the Google platform. There are a wealth of services available on the platform, that the communitiy can put to use when providing access to data and other resources.

Created a Demo Portal and API

An important part of moving the human services API specification for me was to have an actual working demo I could use to better understand what is possible from a technical perspective. Without a functional version of the API specification it is easy miss many of the finer details involved with deploying, managing, testing, monitoring, and generally operating an API in the real world.

I have developed a prototype API in PHP, using the Slim API framework, giving me a working API that I can change and keep in sync with each version of the API specification. The API prototype is available on Github. It is not production ready, so use at your own risk. If you are going to fork make sure and ping me, as I probably have a more recent version available that I have not updated because of time.

I have also developed a working developer portal where I have published the working API prototype. The API portal runs 100% on Github, using Github Pages as the host, and the site using Jekyll as it's CMS. The portal is meant to eventually be forked by any single API implementation, and converted into a custom portal. The portal intentionally uses Twitter Bootstrap, so that anyone can easily customize, while keeping things following a common standard.

Develop Open Tooling

In addition to the platform projects, and the PHP demo, we are regularly investing in new open source tools, as well has help seed and encourage new projects developed on top of the Human Services Data API (HDSA), help service providers better serve their consistuents.

Heroku

Ohana is the original implementation of the Human Services Data Specification (HSDS), which began as a Code for America project. The project is still an active open source project with a number of forks and implementations. Ohana is a Ruby implementation, that runs on Heroku, but can be deployed anywhere.

Open Referral Playground

An open source tool developed by Chris Spilio, for validating the Open Referral schema, providing a dynamic registry of resource types based on latest schema, a visualization of a resource schema - this includes fields, keys, types, formats, etc., download sample CSV templates of resources, and upload and validation of a sample CSV file. A great example of the Open Referral community in action.

  • Github

Human Services Jekyll

This is a prototype for running a Human Services Data API (HSDA) 100% on Github, using Github Pages, and Jekyll as the content and data management system. It is an ongoing work in progress, but would like to keep playing with a free, public, lite website edition that anyone can run as a low cost implementation. I'm also playing around with managing the data using Google Spreadsheet, which syncs with the YAML in the Github repository.
